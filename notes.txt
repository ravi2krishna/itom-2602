    -> SNMP ?? 

        -> Simple Network Management Protocol 

            -> SNMP is a Protocol(Rules) for "Monitoring" and "Managing"
            Network connected devices (Routers, Switches, Servers, IOT etc) in IP Networks.
            It operates over UDP to track Network performance, detect faults and configure remote devices.

            -> SNMP Port 161 / 162

    -> HTTP/S ?? 

        -> Hyper Text Transfer Protocol (Secured)

            -> For "transmitting web data" over World Wide Web (WWW)

                -> https://www.hdfc.bank.in/

            -> HTTP/S Ports 80 / 443 


    -> SSH ?? 

        -> Secure Shell 

            -> For Encrypted Communication between a Client & Server over Insecure Network 

            -> Remote Logins 

            -> SSH Port 22

    -> CMDB (Configuration Management Database)   

        -> https://www.servicenow.com/in/products/it-operations-management/what-is-cmdb.html


        -> A configuration management database (CMDB) is a centralized file that functions as a 
        comprehensive data warehouse, organizing information about an IT environment. 
        CMDB clarifies the relationships between hardware, software components, and 
        networks for improved configuration management.

        -> "CMDB" is Heart Of ServiceNow ITOM 

        -> CMDB is ServiceNow's Master Inventory of everything in your IT Environment.

            -> Every Server, Applications, Database, Network Components, Business Service 
            all of these lives in CMDB as a record.

        -> Use Case 

            -> customer reports an email service is down 

                -> Email Service is an CI 

                -> Email Client Application is an CI 

                -> Email Server (Hardware) is an CI 

        
        -> NOTE: Based on above scenario, CMDB answers questions like 
        
            -> What do we have 

            -> Where they are 

            -> How Does everything connect ?

        -> NOTE: CMDB is the single source of truth for all of ITOM Process
        

        -> CMDB is a centralized repository that stores information about 
        all your IT Assets(Configuration Items) and their relationships in your organization.



    -> Configuration Items (CI's)

        -> A Configuration Item in servicenow is a digital record within CMDB 
        representing any IT Component such as server, Application, service etc. 

        -> Configuration Items (CI's) are "Atoms Of CMDB"

        -> CI's track Infrastructure details (Manufacturer, Vendor, Locations) to support(ITIL Process)


    -> NO CMDB 

        -> https://www.lansweeper.com/wp-content/uploads/2019/06/All-servers-by-type.png

    -> With CMDB 

        -> https://servicenowguru.com/wp-content/uploads/2024/05/bsmMap.jpg


    -> Every Configuration Item (CI) has three things 

        -> Class (What type of Asset it is)

        -> Attributes (Properties)

        -> Relationships (How it connects to other CI's)


    -> Relationships  

        -> Relationship in servicenow CMDB is a connection between 
        multiple configuration Items, that defines 

            -> How they interact with each other 

            -> How they depend on each other 

                -> Runs On 

                -> Used By 

                -> Depends On 

                -> etc 

            -> Runs On (An Application runs on a server)

            -> Depends On (A Service Depends on an Application)

            -> Connected By (A Server is connected by a Switch)

        
        -> NOTE: All relationships are Automatically Discovered using Discovery Process 



    -> Memory Map for all above Concepts 

        -> CMDB (Think of it as a City Map)

        -> CI's (Buildings within the City - Unique Address, Color [Class & Attributes] )

        -> Relationships (Roads Connecting Buildings) 

        -> Discovery Process (Google Maps Vehicles, Keeps Maps Updated)

        -> Service Mapping (Draws Routes between districts (Services))

        -> Event Management (Monitors - Buildings on fire, sends an SOS to fire Department)

    -> ServiceNow Discovery (Automated CI Population for CMDB)

        -> How ServiceNow Discovery Works - Step By Step

            -> Step - 1
            
                -> Schedule & Trigger 

                    -> Define IP Ranges (10.0.0.99/32) -> Only One Address / Device

                    -> Define IP Ranges (10.0.0.0/20) -> 4096 Address / Device

                    -> Define IP Ranges (10.0.0.0/24) -> 256 Address / Device

                    -> Define Schedule 

                        -> Discovery Runs Automatically Every 24 Hours 

                        -> Run Discovery On Demand when needed 

            -> Step - 2

                -> Scan the network 

                    -> MID Server Probes the Given Network using 
                    SNMP, WMI, SSH to find devices 

            
            -> Step - 3

                -> Classify & Identify 

                    -> Probes Collect All Info Like 
                    OS, Hardware, Software etc 

                    -> Sensors Classify Each Type 

            -> Step - 4 

                -> Populate CMDB 

                    -> CI's are Created or Updated in the CMDB 

                    -> Relationships between assets are mapped Automatically

        -> NOTE:

            -> Discovery = Zero Manual Data Entry for Infrastructure CI's 

                -> No Spreadsheets - No Stale Data - CMDB Stays accurate Automatically - Every Scan updates what's changed

    -> Credentials 

        -> Once the MID Server reaches your devices on the network, it needs to 
        log into Servers(Web Servers, App Servers, Database Servers, Caching Servers etc) 
        to interrogate them, that's what Credentials are used for. 

        -> Credentials are Authentication Secrets, stored securely in ServiceNow 
        and MID Server uses these Credentials to access devices.

        -> Credentials are stored in "discovery_credentials" table.

        -> Credentials are never sent in plan text format, they are "Encrypted" using ServiceNow Encryption. 

        -> MID Server receives Encrypted Credentials and decrypts it locally at moment of use 

        -> There are several types of Credentials 

            -> SSH Credentials are used in Linux & Unix Environments 

                -> SSH Credentials are made up of 

                    -> Username
                    -> Public Key 
                    -> Private Key 

                -> Uses Password less Authentication (Way Secure then Password based Authentication)


            -> Windows Credentials are made up of Username & Password 

            -> JDBC Credentials (Database Authentication)

            -> VM Ware (VM Ware Credentials)

            -> Cloud Infrastructure Like AWS, Azure etc use API Keys & Tokens 



    -> Probes & Sensors

        -> Probes & Sensors are two halves of Same Job.

        
        -> Probes

            -> A Probe Is a Question 

                -> It's a Script / Command that MID Server executes against target devices to collect RAW Data 

                -> It Runs on MID Server Side, Reaches to the target device and brings back all the 
                raw output like commands, XML, JSON Data, WMI Queries etc  

                -> The raw output will be stored at ServiceNow side inside 
                ECC Queue (External Communication Channel)

       
        -> Sensors 

            -> A Sensor Is an Answer          

                -> It's a Script(javascript) that takes raw output from the Probe 
                and pareses it, extracting fields like OS, IP, Hostname, Ports etc 
                and maps them to CI Attributes, updating as records in CMDB.


        -> Detective Story 

            -> Detective Assistant / Probe 

                -> Goes to the Crime Scene, takes photos, lift fingerprints, collects samples etc 
                collection of raw evidence 

                -> ITOM: Probes Runs Scripts, Collects Data 

            
            -> Bag Of evidence / ECC Queue 

                -> Bag of evidence contains full of unprocessed raw evidence items 

                -> ITOM: Raw Data Collected i.e unprocessed data 

            -> Detective / Sensor 

                -> Detective sits at desk, opens the bag of evidence, 
                examines each item, then extracts meaning of these items 
                and finally writes case report 

                -> ITOM: Extracts data from ECC Queue and gives meaningful fields 
                like OS, IP, Hostname, Ports etc 


        -> NOTE 

            ServiceNow Says -> Go Discover this IP 
                        |
            MID Server Sends PROBE -> Runs Scripts on target device
                        |
            Raw data drops into ECC Queue (ecc_queue table)
                        |
            Sensor picks it up -> Reads, Parses & Extracts 
                        |
            CMDB Gets Updated With NEW / UPDATES CI Records


    
    -> Patterns

        -> Patterns are ServiceNow's Modern Replacement for Older Probes/Sensors 
        approach for Applications and Software Discovery 

        -> Still Proves and Sensors are heavily used for Infrastructure
        (OS, IP, Hardware like CPU, MEMORY, DISK etc)

        -> NOTE: Patterns are preferred method for Discovering Applications & Softwares 

        -> A Pattern is a Declarative Definition, Unlike Imperative Definition Of Probes & Sensors.

        -> Imperative Definition means "HOW TO DO"

        -> Declarative Definition means "WHAT TO DO"

        -> NOTE: Patterns are written in Visual Designer in ServiceNow (Pattern Designer)
        rather then raw javascript 

    

    -> Classification

        -> CI Classification is the final step, the "Classifier Engine" or "Decision Engine"
        determines what CI Class to create and whether to create a new CI or update existing CI
        
        -> When a Sensor or Pattern finished parsing data from devices, it hands over 
        raw attributes to CI Classifier Engine 

        -> The Classifier's Job is to look at those attributes and answer two questions 

            -> What Type Of CI is this ? 

            -> Does it already exist in the CMDB ? 

        
        -> NOTE: Classifier Engine == Identification Rules 

        -> Identification / Reconciliation 

            -> Identification Rules for matching configuration items before insert/update (adding/updating)

    -> IRE (Identification and Reconciliation Engine )

        -> IRE is underlying ServiceNow Engine that uses identifier rules 

            -> Each class has a defined set of "identifier fields" that together makes a CI Unique 

            -> "Criterion Attributes" these are like "Unique fingerprints" of a CI 

        -> For a web server like Apache, the combination of  
        sys_class_name + config_file + version

        -> For a Docker Container, container_id is the unique identifier 

        -> Simple Analogy 

            -> Think of IRE like an Hospital Reception Desk 

                -> Patient Walks In (Discovery Finds a device)

                -> Receptionist asks: Have you been here before ?

                    -> YES: Pull Exiting File, and update it (Reconciliation)

                    -> NO: Create a new patient file (New CI Created)

        -> Discovery Finds Device 

        -> Patterns/Sensors collects Attributes 

        -> IRE receives Attributes 

            What type is this CI <- CI Classification 

            Does this CI already exist ? <- Identification

                YES                         NO
            Update CI                   Create CI
            (Reconciliation)            (New Record)
